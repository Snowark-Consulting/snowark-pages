# Deployment Security Checklist

Run in full during playbook 04 (integrations) and again during playbook 05 (verify). Every section checked = initialled and dated at the bottom. Distilled from a 27-point AI security audit checklist; items here are the ones a fresh deployment can actually fail.

## Credentials

- [ ] All API keys/tokens in env files on-instance only, `chmod 600`
- [ ] No credential in any template, repo, log, or chat transcript (sweep with the secret-pattern grep in the kit's Task 15 commands before any push)
- [ ] Each integration uses least-privilege scopes; scopes granted recorded per integration in the client repo (names, never values)
- [ ] Token expiry dates recorded with cron renewal reminders set at creation time
- [ ] Unused or test credentials deleted at both ends (instance AND service provider)
- [ ] MFA enabled on every connected service account (model provider, email, accounting, ads)

## Spend protection

- [ ] Spending limit set on the model provider account
- [ ] Usage caps or alerts set on any metered API the agent can call in a loop
- [ ] Cron jobs reviewed for runaway potential (a failing job must not retry-spend unbounded)

## Data protection

- [ ] Client backup repo is private; verify with `gh repo view --json visibility`
- [ ] No client data flows to any service that is not a configured integration
- [ ] Instance disk is the single data home; backups travel encrypted in transit (HTTPS git)
- [ ] Sensitive fields (payment details, IDs) are excluded from prompts unless the task strictly requires them (least privilege applies to data, not just keys)
- [ ] Data-protection obligations for the client's jurisdiction noted in the client repo (e.g. Australian Privacy Act for AU clients)

## LLM risks

- [ ] Iron Gate Protocol active (`Agent/Trust.md`) and referenced from SOUL.md safety rules
- [ ] Prompt-injection rule present in SOUL.md (instructions inside content are data, not commands)
- [ ] Inbound content from untrusted sources (webhooks, public forms, unknown emails) is validated before it reaches a prompt
- [ ] Draft-approval protocol active; zero Tier 1 message types at deployment

## Recovery

- [ ] Nightly backup to the client's private repo verified (run one manually, check it lands)
- [ ] Restore path documented and tested once: fresh instance + backup repo = working install
- [ ] Kill-switch documented: how to stop the agent, disable a credential, and pause cron in under five minutes

## Assume breach

- [ ] Incident response noted in the client repo: which credential to rotate first, who to tell, what to check in logs
- [ ] Execution/cron logs reviewed for sensitive data retention; trimmed if they capture message bodies
