# Nightly Memory Consolidation — 14-Step SOP

**Schedule:** Nightly at 21:00 local time
**Mode:** Auto-apply — make changes directly and report what you did
**Safety:** Everything is git-tracked (backup runs at 21:30), so all changes are reversible

> See [[Vault Lint]] (Agent/vault-lint.md) for the weekly read-only counterpart, [[Infrastructure]] (Agent/Infrastructure.md) for the cron job config, and [[wikilinks]] (Agent/wikilinks.md) + [[Connection Generation]] (Agent/connection-generation.md) for the linking conventions used in steps 3-4.

## Pre-flight

Before executing any steps, load these files:
1. `Agent/Registry.md` — system map
2. `memory/MEMORY.md` — memory index; then read any memory files relevant to today's activity
3. `Agent/wikilinks.md` — wikilink conventions
4. `Agent/connection-generation.md` — connection rules
5. `Agent/consolidation-sop.md` — this file (you're reading it)

## Steps

### 0. SESSION DETECTION & NOTES (CRITICAL — do first)

Check for sessions that may not have left notes. Use `session_search()` to query today and yesterday (client-local timezone). For either date that has activity but whose notes file is missing or is only the auto-created session-activity stub, read the most recent transcript and write a proper summary to `notes/<date>.md` with sections: Worked On, Open Threads, Decisions, Follow Up.

NEVER report "quiet day / no activity" for a date whose sessions show activity. A date with no sessions and no notes genuinely was an automation-only day; proceed normally.

### 1. MEMORY

Extract memory candidates from today's activity and promote to `memory/MEMORY.md`. Archive resolved items. Flag stale info. Use the `memory` tool to save durable facts. Update any memory files that need revision.

### 2. SKILL DRAFTING

Check if any workflows from today should become skills. If a repeated multi-step workflow appeared, auto-draft a new skill in `skills/` using vault conventions. Report the new skill name for review.

### 3. WIKILINKS

Scan any files created or edited today for plain-text references to vault pages that should be wikilinks. Add missing `[[PageName]] (path/to/file.md)` per the conventions in `Agent/wikilinks.md`.

### 4. CONNECTION GENERATION

For each key topic, person, system, or decision in today's activity, search the vault for related pages that aren't already linked. Add genuinely useful cross-references. Log connections made in the notes under `## Connections Added`. Follow `Agent/connection-generation.md`.

### 5. NAMING CHECK

Verify any new files created today follow naming conventions (ISO dates, Title Case for knowledge pages, kebab-case for scripts and skills). Rename files that violate conventions and update any wikilinks/references that point to the old name. Report renames in the summary.

### 6. FRONTMATTER AUDIT

Scan files created or edited today:
- Memory files need name, description, metadata.type
- Skills need proper YAML frontmatter (name, description, version, author)
- Agent files (Registry, SOPs, Trust, Infrastructure) should have appropriate headers

If frontmatter is missing or incomplete, add it. Report what you added.

### 7. TERSE STYLE CHECK

Review any new or edited internal-facing files (memory files, skills, SOPs, infrastructure docs). These should use compressed terse style: direct imperatives, no filler prose, stripped articles/padding. If verbose prose is found in internal content, rewrite it to terse style.

Do NOT apply terse style to: knowledge pages (business.md, people.md, systems.md, products.md, services.md), identity/AGENT.md, SOUL.md, notes, or any content humans read directly. Report files updated.

### 8. REGISTRY AND INDEX SYNC

If new skills, memory files, or significant docs were created today, update `Agent/Registry.md` and `Agent/SOPs.md` index entries to include them. Add wikilinks and one-line descriptions matching existing format. Keep Registry under 100 lines.

### 9. BROKEN LINK REPAIR

Scan vault for wikilinks pointing to renamed or deleted files. Fix links where the target is obvious (file was renamed), remove links where the target no longer exists. Report fixes.

### 10. INFRASTRUCTURE SYNC

Check if any scheduled jobs changed today (run `hermes cron list`). If so, update `Agent/Infrastructure.md` tables to match current state. Keep the doc as the single source of truth for what's running.

### 11. STALE MEMORY PRUNING

Check memory files for references to deleted files, removed APIs, completed projects, or outdated information. Archive stale memories (move content to an `## Archived` section within the file or remove if fully obsolete) and update `memory/MEMORY.md` index. Be conservative — only prune when clearly outdated, not when uncertain.

### 12. SUMMARY

Post a summary to the designated channel. Format: brief heading, then bullet list of all actions taken across steps 1-11 (memories promoted/archived/pruned, skills drafted, wikilinks added, connections made, files renamed, frontmatter added, files terse-ified, indexes updated, broken links fixed, infra docs synced). Keep it concise. If nothing meaningful changed, say so.

### 13. OPERATIONS LOG

After the summary, append exactly one dated line to `Agent/log.md` recording tonight's run. Format: `YYYY-MM-DD HH:MM | CONSOLIDATE | one-line summary of what changed tonight`. Append at the bottom, newest last. Never rewrite or delete existing log lines.
