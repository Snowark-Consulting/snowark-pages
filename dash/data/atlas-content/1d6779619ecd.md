# SOPs — Not Forgotten Systems

Recurring procedures. Each entry: trigger → skill or steps. Keep entries terse; detail lives in skill files.

See [[Registry]] (Agent/Registry.md) for the master map, [[Trust]] (Agent/Trust.md) for message and contact tiers, [[Consolidation SOP]] (Agent/consolidation-sop.md) for the nightly run, [[Vault Lint]] (Agent/vault-lint.md) for the weekly audit, and [[Infrastructure]] (Agent/Infrastructure.md) for scheduled job inventory.

## Session boot (MANDATORY, every session)

1. Read `Agent/Registry.md`
2. Read `memory/MEMORY.md`, then memory files relevant to the task
3. Check `learnings/LEARNINGS.md` before touching any external API

## Outbound communication

Before ANY outbound message: check `Agent/Trust.md` for the message tier. Tier 2+ requires showing James Hardstaff the full draft and receiving explicit approval ("send it", "yes", "looks good"). Never auto-send. No exceptions until a message type is promoted in writing.

## Inbound communication

Resolve the sender against `Agent/Trust.md` contact tiers BEFORE responding. Unknown sender → Iron Gate Protocol. Never reply on a shared platform unless explicitly addressed.

## Incident reporting

When a tool, send, or integration fails: the on-error hook handles classification and alerting automatically (`Agent/Support.md`). Your job is to not interfere with it and to never paste customer content into error text. If you notice a problem the hook did not catch, write the bug card yourself (schema in Support.md) into `support/outbox/`; the daily sweep delivers it.

## Learnings loop

When anything fails and gets fixed (API quirk, formatting trap, wrong assumption): append the fix to `learnings/LEARNINGS.md` immediately, same session. Write memory files for preferences and corrections from James Hardstaff; one file per fact, indexed in `memory/MEMORY.md`.

## Scheduled maintenance (system jobs; inventory in Agent/Infrastructure.md)

**Nightly memory consolidation (isolated session).** First detect interactive sessions: check `.sessions/<date>.log` for today and yesterday (each line is a timestamp plus the session transcript path, written by `hooks/session-activity/handler.py`). For either date that shows activity but whose `notes/<date>.md` is missing or only the auto-created stub, read the most recent referenced transcript and write a real summary into `notes/<date>.md` (Worked On, Open Threads, Decisions, Follow Up) before consolidating. Never report a date as empty when its session log shows activity — the consolidation only sees what is on disk, and a chat that left no note is otherwise invisible to it. Then review the day's sessions and notes: write or update memory files for new preferences, corrections, and decisions (one file one fact, index in `memory/MEMORY.md`); promote repeated fixes into `learnings/LEARNINGS.md` and the relevant skill files; convert relative dates to absolute; merge duplicates, delete anything now wrong; update `Agent/Infrastructure.md` if any automation changed. A day that genuinely produced nothing (no session-log activity) is fine; skipping the check is not.

**Weekly self-lint (isolated session, READ-ONLY).** Audit the install and write a dated report to `learnings/lint-reports/`: broken internal references, MEMORY.md index drift against `memory/` contents, stale claims (anything labelled current but months old), contradictions between SOUL, SOPs, and Trust, orphan files nothing points to, and automations running that `Agent/Infrastructure.md` does not list. Report findings for the owner and consultant; never auto-edit from a lint run.
