# Integration checklists

One file per platform. Each follows the same skeleton:
**Prerequisites** (accounts, access the client must provide) → **Credential setup** (what to create, scopes, where it lives on-instance per `core/security/secrets-pattern.md`) → **Configuration** (files/env to write) → **Verification** (a command or action proving it works) → **Known traps** (append as learned; this is where deployment learnings land).

Only integrations named in `client.yaml` get executed during playbook 04.

General rule for all integrations: high-frequency polling jobs (inbox sweeps, status checks) run in isolated sessions, not the main agent session. Loading full boot context 20+ times a day for a utility poll burns tokens for nothing.

Stronger rule for the inbox sweep specifically: it polls every 2 minutes, so it MUST sit behind the `Agent/inbox-pulse.sh` cost gate — a cheap unread check that only spawns a session when mail is actually waiting (see `integrations/email.md` → Inbox pulse). The same gate pattern is the right move for any sub-5-minute poll: check cheaply first, spend tokens only on a real hit.
