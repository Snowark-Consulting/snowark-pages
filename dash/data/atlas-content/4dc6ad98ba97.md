# Integration — Email

> See [[gmail]] (learnings/cards/gmail.md) for API traps, [[Infrastructure]] (Agent/Infrastructure.md) for running watchdog, and [[LEARNINGS]] (learnings/LEARNINGS.md) for cross-pollinated fixes.

**Prerequisites:** an inbox for Snowark (Google Workspace or M365); admin access held by Josh.

**Credential setup:** OAuth client (Gmail API) or an app password (generic IMAP). Store on-instance in the agent env per `core/security/secrets-pattern.md`, `chmod 600`. Never in any repo.

**Configuration:** set `PULSE_MODE` (`gmail_api` or `imap`) and inbox creds in the agent env; wire `Agent/inbox-pulse.sh` as the cheap unread pre-check ahead of any sweep.

**Verification:** run `Agent/inbox-pulse.sh` once with no unread mail — it exits without spawning a session; send a test mail and confirm it spawns exactly one.

**Known traps:**

- **The pulse must FAIL-CLOSED** on a check error (skip + log, never spawn a runaway session).
- **`gog` hyphen trap:** `gog gmail mark read <id>` (space) silently fails with exit code 2. The correct syntax is `gog gmail mark-read <id>` (hyphen). An agent won't notice the failure unless exit codes are explicitly checked. Always use `mark-read`, never `mark read`.
- **Thread ID ≠ message ID:** The Gmail API (and `gog`) can return thread IDs that match only the first message in a thread. If that message was already read, `mark-read` on the thread ID does nothing, and subsequent unread messages in the same thread remain unread forever. Always resolve to individual `unread_msg_ids` and mark those, not the thread ID.
- **Idempotency guard:** The poll loop must track which message IDs it has already processed, so a mark-read failure doesn't cause the same messages to be summarised and reported on every tick.
- **Gmail `newer_than:15m` = 15 MONTHS:** Gmail's `newer_than:` operator only accepts `d` (days), `m` (MONTHS), `y` (years). There is no minute unit. Never use `newer_than` for sub-day windows — filter by `internalDate` in code after fetching.
- **Gmail batchModify returns 204 empty:** `POST /messages/batchModify` returns HTTP 204 with an empty body. `json.load()` on empty string crashes with `Expecting value`. Check body before parsing: `raw = r.read(); return json.loads(raw) if raw else {}`. Verify by re-fetching labels, not by retrying.
